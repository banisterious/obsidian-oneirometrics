# UniversalWorkerPool & Chart Loading Issues Investigation

## Branch: `fix/scrape-issues`

## Problem Summary

The UniversalWorkerPool is experiencing constant worker failures, causing scraping to fall back to legacy systems and charts to fail loading, showing placeholders instead.

## Issues Identified

### 1. UniversalWorkerPool Worker Failures

**Symptoms:**
- Workers fail immediately after creation (within milliseconds)
- Infinite loop: Create → Error → Mark Unhealthy → Recreate → Repeat
- Pattern observed in logs: `oomp-logs-20250613-173144.json`

**Log Pattern:**
```json
{
  "timestamp": "2025-06-14T00:31:44.025Z",
  "level": "info",
  "category": "UniversalWorkerPool:Worker",
  "message": "Worker recreated: worker-1"
},
{
  "timestamp": "2025-06-14T00:31:44.024Z", 
  "level": "warn",
  "category": "UniversalWorkerPool:Worker",
  "message": "Worker marked as unhealthy: worker-1"
},
{
  "timestamp": "2025-06-14T00:31:44.024Z",
  "level": "error", 
  "category": "UniversalWorkerPool:Worker",
  "message": "Worker error: worker-1"
}
```

**Root Cause:**
The worker script generated by `getUniversalWorkerScript()` is extremely complex (~900 lines) and contains:
- Multiple task processors (date filtering, metrics calculation, sentiment analysis)
- Complex metrics calculations with advanced algorithms
- Time-based metrics processing
- Content analysis functions
- Aggregation logic

**Likely Failure Points:**
1. **Syntax Errors**: Complex string concatenation may introduce syntax issues
2. **Missing Dependencies**: Worker context doesn't have access to main thread functions
3. **Memory Issues**: Large script size may exceed worker memory limits
4. **Runtime Errors**: Complex initialization logic failing silently

### 2. Chart Loading Failures

**Symptoms:**
- Charts show placeholder: "📊 Chart Data Not Available"
- Placeholder message: "Charts will appear here after running a metrics scrape"
- Charts fail to load even after successful scraping

**Flow:**
1. UniversalMetricsCalculator attempts scraping with worker pool
2. Worker pool fails → Falls back to legacy MetricsCollector
3. Legacy system completes successfully
4. Chart generation still fails → Shows placeholder

**Root Cause:**
Chart loading depends on successful worker pool operations, but even when legacy fallback works, the chart data pipeline is broken.

### 3. Scraping Fallback Issues

**Current Flow:**
```typescript
// main.ts scrapeMetrics()
try {
  // UniversalMetricsCalculator with 2-minute timeout
  await universalCalculator.scrapeMetrics();
} catch (error) {
  // Fallback to legacy MetricsCollector
  await this.metricsCollector.scrapeMetrics();
}
```

**Issues:**
- 2-minute timeout may be too long for failing workers
- Fallback works but chart pipeline still broken
- No proper error reporting to user about worker failures

## Technical Analysis

### UniversalWorkerPool Architecture Issues

1. **Overly Complex Worker Script**
   - Single worker handles 15+ different task types
   - Complex metrics calculations in worker context
   - Large script size (~900 lines of generated code)

2. **Error Handling Problems**
   - Workers fail silently with generic "Worker error" messages
   - No detailed error information captured
   - Infinite recreation loop with no circuit breaker

3. **Resource Management**
   - No memory limits enforced
   - No script size validation
   - No worker initialization timeout

### Chart Loading Pipeline Issues

1. **Dependency Chain**
   - Charts depend on successful worker pool operations
   - No fallback chart generation from legacy data
   - Cache invalidation issues

2. **Error Recovery**
   - No graceful degradation when workers fail
   - Placeholder shown instead of attempting alternative chart generation

## Proposed Solutions

### Phase 1: Immediate Fixes

1. **Add Worker Error Debugging**
   - Capture detailed worker error messages
   - Add worker script validation
   - Implement circuit breaker pattern

2. **Reduce Worker Complexity**
   - Split complex worker into smaller, focused workers
   - Remove advanced metrics from worker context
   - Simplify task processors

3. **Fix Chart Loading Fallback**
   - Enable chart generation from legacy scraping data
   - Add proper error handling in chart pipeline
   - Implement chart data validation

### Phase 2: Architecture Improvements

1. **Worker Pool Redesign**
   - Implement dedicated workers for specific task types
   - Add worker health monitoring with detailed diagnostics
   - Implement proper resource limits

2. **Enhanced Error Recovery**
   - Add circuit breaker pattern for failing workers
   - Implement graceful degradation strategies
   - Add user-friendly error reporting

3. **Chart System Robustness**
   - Decouple chart generation from worker pool
   - Add multiple chart data sources
   - Implement chart caching improvements
